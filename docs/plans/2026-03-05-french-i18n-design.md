# French i18n — Design Doc

**Date:** 2026-03-05
**Status:** Approved
**Approach:** Client-side swap — English in HTML, French in JSON

## Context

The website is 8 vanilla HTML pages deployed to GitHub Pages. All content is hardcoded in English. No build step, no framework. The movement is based in Brussels, so a significant chunk of visitors speak French.

## Goal

Add French translation without changing the English-first writing workflow. French is a non-invasive option: detected via browser language, offered via a one-time toast, remembered with a cookie.

## Constraints

- No build tools, no framework, no new dependencies
- English remains the natural content in HTML — not extracted to JSON
- French is derived, not co-authored — if a French key is missing, English stays
- No SEO requirement for French (audience finds the site through the movement, not Google)

## Design

### 1. Content keying

Every translatable text element gets a `data-i18n` attribute with a namespaced key.

```html
<h1 data-i18n="index.hero.title">Grassroot Hoppers</h1>
<p data-i18n="index.hero.subtitle">The Transition Towns of digital infrastructure</p>
```

Key format: `{page}.{section}.{element}` — e.g., `index.slide02.heading`, `social.slide03.body`.

English content stays in the HTML. It's the source of truth and the fallback.

### 2. French translation file

**File:** `website/i18n/fr.json`

Flat key-value map:

```json
{
  "index.hero.title": "Grassroot Hoppers",
  "index.hero.subtitle": "Le Transition Towns de l'infrastructure numérique",
  "index.slide02.heading": "La barrière technique s'effondre"
}
```

Missing keys = English stays. Translation can be incremental.

### 3. Language detection, toast, and swap

**File:** `website/i18n/i18n.js`

Three responsibilities:

1. **Check cookie.** If `lang=en` or `lang=fr` exists, respect it. No toast.
2. **Detect language.** If no cookie, check `navigator.languages` for any `fr` variant (`fr`, `fr-BE`, `fr-FR`). If found, show the toast.
3. **Show toast once.** Small banner at the bottom: *"Ce site est aussi disponible en français"* with two buttons: *"Passer en français"* / *"Non merci"*. Either choice sets the cookie and dismisses the toast.

If the user picks French (or has `lang=fr` cookie):
- Fetch `i18n/fr.json`
- Swap `textContent` of all `[data-i18n]` elements where a key exists
- Set `<html lang="fr">`

Toast is styled inline in JS to match the site's dark/warm aesthetic.

### 4. Manual language toggle

A small "FR / EN" link in the footer of every page. Clicking it sets the cookie and swaps language. Always available, never prominent.

### 5. Per-page changes

Each of the 8 HTML files gets:
- `data-i18n` attributes on all translatable text elements
- `<script src="i18n/i18n.js"></script>` before `</body>`

### 6. New files

| File | Purpose |
|------|---------|
| `website/i18n/fr.json` | French translations (flat key-value) |
| `website/i18n/i18n.js` | Detection, toast, swap logic |

### 7. What doesn't change

- No new dependencies
- No build step
- No framework
- No duplicate HTML files
- English writing workflow is untouched

## Translation workflow

1. Write/edit English copy in HTML as usual
2. If you add a new `data-i18n` key, add the French translation to `fr.json`
3. If you forget, English shows — nothing breaks
4. Initial `fr.json` can be generated by extracting all keys and translating in bulk (AI, human, whatever)
